# 「吃了么」饮食记录微信小程序 产品需求文档（PRD）

**文档版本**：V1.0**适用场景**：个人健康饮食管理、好友群组监督、膳食营养分析、AI 食物识别**核心定位**：轻量化微信原生饮食打卡小程序，支持个人记录、营养分析、社交群组、拍照分享全流程**技术方案**：微信云开发 CloudBase 全栈方案（零服务器、零备案、快速上线、平滑迭代）

---

## 一、文档概述

### 1.1 项目背景

基于微信生态打造轻量化饮食管理工具，解决用户三餐记录不便、营养摄入不清晰、缺乏监督陪伴的痛点，支持拍照留证、AI 自动识别食物与营养成分，同时通过群组功能实现好友间饮食监督、打卡分享，兼顾实用性与社交属性。

### 1.2 文档适用范围

本文档涵盖**产品全部功能需求、业务规则、数据结构、存储方案、推荐技术路线、开发规范、权限规则、迭代规划**，可直接用于小程序开发、功能验收、后续迭代扩展。

### 1.3 核心用户群体
- 健身增肌 / 减脂人群、控糖控重用户
- 需要饮食监督、结伴打卡的用户
- 日常健康管理、记录饮食习惯的普通用户

---

## 二、整体产品目标与核心价值

### 2.1 核心目标
1. 实现极简的一日三餐 + 加餐饮食记录能力，降低用户操作成本
2. 接入 AI 图像识别，自动匹配膳食营养成分，无需手动录入
3. 搭建轻量化好友群组体系，实现饮食记录共享、互相监督
4. 全流程依托微信生态，无需下载、一键登录、一键分享

### 2.2 产品核心价值
- 记录价值：标准化饮食台账，可追溯、可查看历史数据
- 营养价值：自动核算每餐热量、碳水、蛋白质、脂肪等核心膳食指标
- 社交价值：群组结伴打卡，解决个人坚持难的问题
- 便捷价值：微信原生体验，拍照即记录，无需复杂操作

## 三、核心功能需求（完整全量梳理）

本部分包含你提出的**全部需求**，分模块清晰定义，无遗漏、无歧义。

### 3.1 用户基础模块（必选核心）
1. **微信授权一键登录**
    - 无需注册、无需手机号，通过微信 openid 唯一标识用户
    - 自动获取微信昵称、头像，完成用户信息初始化
    - 支持游客模式预览，登录后同步数据
    
    
2. **个人信息管理**
    - 支持昵称、头像修改
    - 支持隐私设置、记录默认权限设置
    
    
3. **账号数据隔离**
    - 单用户数据独立存储，默认仅本人可见
    - 支持主动开放权限至指定群组

### 3.2 饮食记录核心模块（必选核心）
1. **餐次类型定义**
    - 固定分类：早餐、午餐、晚餐、加餐
    - 支持单条记录绑定单一餐次，支持日期自由选择
    
    
2. **记录编辑能力**
    - 文字备注：记录饮食内容、饮食感受、特殊说明
    - 多图上传：支持拍照、本地相册多选上传，单条记录最多 9 张图
    - 时间戳：自动记录创建时间，支持修改记录日期
    - 支持记录的新增、编辑、删除、历史查询
    
    
3. **历史记录查看**
    - 日历视图：按日期筛选当日全部饮食记录
    - 列表视图：按时间倒序展示全部历史记录
    - 支持按餐次、日期筛选查询

### 3.3 膳食营养与 AI 识别模块（迭代核心）
1. **膳食成分标准化记录**
    - 核心指标：总热量（kcal）、碳水化合物（g）、蛋白质（g）、脂肪（g）
    - 扩展指标：膳食纤维（g）、钠（mg）
    - 支持单条记录手动填写、修改全部营养指标
    
    
2. **AI 食物图像识别**
    - 用户上传菜品照片后，一键触发 AI 识别
    - 自动识别食物名称、品类，置信度匹配
    - 自动匹配对应食物的标准营养成分数据，回填至记录
    - 支持 AI 结果修正，手动修改覆盖 AI 数据
    
    
3. **营养数据统计**
    - 当日营养汇总：总热量、三大营养素单日总量、占比
    - 支持按日、按周查看营养摄入趋势

### 3.4 分享传播模块（必选核心）
1. **单条记录分享**
    - 支持分享至微信好友、微信群聊
    - 自动生成饮食记录卡片，包含餐次、内容、图片、营养数据
    
    
2. **全局分享能力**
    - 支持小程序首页分享、群组邀请分享
    - 分享卡片携带参数，支持跳转对应记录、对应群组

### 3.5 好友群组模块（社交核心）
1. **群组基础管理**
    - 用户可自由创建群组，自定义群名称、群简介
    - 群主拥有管理员权限，可移除成员、解散群组
    - 支持私密群组模式，仅通过邀请可加入，无公开加入入口
    
    
2. **成员邀请与管理**
    - 通过微信分享卡片一键邀请好友进组
    - 好友点击卡片，一键授权加入对应群组
    - 单用户可加入多个群组，无数量限制
    - 成员可自主退出群组
    
    
3. **群组社交与数据共享**
    - 群组动态流：按时间展示群内所有成员的公开饮食记录
    - 群内仅可查看成员**主动授权公开**的记录，不可查看隐私记录
    - 支持单条记录设置「仅自己可见」「指定群组可见」
    - 可选扩展：记录点赞、简短评论互动
    
    
4. **群组数据统计**
    - 展示群内成员当日打卡情况、饮食记录概览

## 四、非功能性需求
1. **兼容性要求**
    - 兼容微信 iOS、Android 全版本客户端
    - 兼容低端机型，图片加载、页面渲染无卡顿
    
    
2. **性能要求**
    - 页面加载速度≤2 秒
    - 图片上传、AI 识别响应≤5 秒
    - 数据库查询无延迟，列表滑动流畅
    
    
3. **存储与安全要求**
    - 用户图片、文字数据永久云端存储
    - 用户隐私严格隔离，无越权查看、越权修改
    - 微信 openid 脱敏处理，不泄露用户隐私信息
    
    
4. **合规要求**
    - 符合微信小程序平台审核规范
    - 符合网络安全、用户数据存储相关法规
    - 不收集无关用户信息，仅使用必要权限

## 五、推荐完整技术路线（微信云开发 CloudBase 最优方案）

### 5.1 技术方案选型结论

**唯一推荐方案：微信云开发 CloudBase 全栈原生方案**完全匹配本项目需求，**零服务器、零域名备案、零后端运维、7 天可上线基础版、无缝支持 AI 识别与群组功能**，是个人开发者、轻量化小程序的最优解，无替代方案更适配。

### 5.2 前端技术栈
- **框架**：微信原生小程序（WXML + WXSS + JavaScript/TypeScript）
- 选型理由：和微信生态深度绑定，登录、分享、云开发调用无兼容问题，包体积小、加载快，开发成本最低
- 备选方案：Uni-app（如需后续同步发布 H5 可切换，原生小程序为首选）

### 5.3 后端 & 云服务技术栈（全托管）
1. **云数据库（NoSQL）**
    - 用途：存储用户信息、饮食记录、营养数据、群组信息、群成员关系
    - 优势：JSON 格式、无需建表、自带权限安全规则、天然支持小程序鉴权
    
    
2. **云存储**
    - 用途：存储用户上传的饮食拍照图片，自动生成 HTTPS 访问链接
    - 优势：自带 CDN 加速、上传限流、安全校验，无需单独配置对象存储
    
    
3. **云函数（Node.js 环境）**
    - 用途：处理业务逻辑、权限校验、AI 识别接口调用、群组关系处理
    - 核心作用：解决小程序前端无法直接调用第三方 AI 接口的跨域、密钥泄露问题，所有敏感逻辑在云端执行
    
    
4. **云调用**
    - 直接对接微信开放平台能力，无需额外配置 access_token，一键实现微信登录、分享、消息能力

### 5.4 AI 识别第三方服务选型
- **核心服务**：百度智能云-菜品识别 API（首选）
- 备选：阿里云食品图像识别、火山引擎菜品识别
- 接入方式：小程序上传图片→云函数携带密钥请求 AI 接口→解析营养数据→回写云数据库
- 优势：现成模型、识别准确率高、按量计费、前期成本极低

### 5.5 技术架构流程图

```plaintext
微信用户 → 微信原生小程序前端
  ↓
微信授权登录 → CloudBase自动获取openid完成鉴权
  ↓
功能操作：
  1. 饮食记录/营养录入 → 写入云数据库
  2. 拍照上传 → 上传至云存储
  3. AI识别 → 触发云函数调用第三方AI接口
  4. 群组/邀请/查看好友记录 → 云数据库+安全规则控制权限
  5. 微信分享 → 微信原生分享能力
  ↓
数据持久化存储，权限隔离管控，前端渲染展示
```

## 六、数据存储方案（明确存储位置与结构）

### 6.1 数据分类与存储位置
1. **结构化业务数据（用户、记录、营养、群组、成员）**
    - 存储位置：**微信云开发 CloudBase 云数据库**
    - 数据格式：BSON（类 JSON），支持索引、联表查询、权限控制
    
    
2. **非结构化媒体数据（用户上传的饮食照片）**
    - 存储位置：**微信云开发 CloudBase 云存储**
    - 规则：图片压缩上传、自动生成 URL、仅授权用户可访问
    
    
3. **临时缓存数据（用户信息、当日记录）**
    - 存储位置：小程序本地缓存（wx.setStorage）
    - 规则：仅做加速使用，不做持久化存储，不存隐私数据

### 6.2 核心数据库集合设计（5 张核心集合，直接可使用）
1. **users（用户信息表）**：openid、昵称、头像、创建时间、隐私配置
2. **food_records（饮食记录表）**：用户标识、日期、餐次、备注、图片列表、AI 识别结果、全量营养指标、可见权限、关联群组 ID
3. **groups（群组信息表）**：群名称、简介、创建人、创建时间、是否私密、成员数量
4. **group_members（群成员关系表）**：群组 ID、用户标识、加入时间、管理员权限
5. **group_interact（群组互动表）**：记录 ID、用户标识、评论 / 点赞内容、创建时间

### 6.3 权限安全规则

通过云数据库安全规则实现，无需编写后端鉴权代码：
1. 用户仅可修改、删除自己的饮食记录
2. 用户仅可查看自己加入的群组内成员的公开记录
3. 仅群主可修改群组信息、移除群成员
4. 禁止越权查询、越权修改非本人、非本群组数据

## 九、风险与应对方案
1. **微信平台审核风险**
    - 应对：功能合规，不涉及敏感内容，遵循微信小程序社交类目规范
    
    
2. **AI 识别准确率不足**
    - 应对：保留手动修改营养成分功能，用户可修正 AI 结果
    
    
3. **用户隐私泄露风险**
    - 应对：云数据库安全规则强制隔离，仅公开权限的内容可被群组查看
    
    
4. **存储与带宽成本上涨**
    - 应对：前期云存储免费额度充足，后期开启图片压缩、限流策略，成本可控

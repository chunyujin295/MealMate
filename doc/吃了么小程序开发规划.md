# 「吃了么」饮食记录微信小程序 开发规划

## 一、技术栈

### 1.1 前端
| 层级 | 技术 | 说明 |
|------|------|------|
| 框架 | 微信原生小程序 (WXML + WXSS + TypeScript) | 与微信生态深度绑定，包体积小，登录/分享/云开发无兼容问题 |
| 状态管理 | 小程序全局 app.globalData + Page data | 轻量级场景无需引入 MobX/Redux |
| UI 组件 | WeUI 扩展库 + 自定义组件 | 微信官方组件库，视觉统一 |

### 1.2 后端 & 云服务（全托管）
| 层级 | 技术 | 说明 |
|------|------|------|
| 云数据库 | CloudBase NoSQL (BSON) | 用户/记录/群组/营养等结构化数据 |
| 云存储 | CloudBase 云存储 | 饮食拍照图片，自带 CDN + HTTPS |
| 云函数 | Node.js (18+) | 业务逻辑、AI 接口调用、群组关系处理 |
| 云调用 | wx-server-sdk | 免 access_token 对接微信开放平台能力 |
| 鉴权 | CloudBase 自动鉴权 | 基于微信 openid，无需自建鉴权系统 |

### 1.3 AI 识别服务
| 服务 | 厂商 | 用途 |
|------|------|------|
| 菜品识别 API | 百度智能云（首选） | 上传食物图片 → 识别菜品名称 → 匹配营养数据 |
| 备选 | 阿里云食品图像识别 / 火山引擎 | 若主服务不稳定可切换 |

### 1.4 工具链
| 工具 | 用途 |
|------|------|
| 微信开发者工具 | 开发调试、云函数上传、云数据库管理 |
| Git + GitHub/Gitee | 版本管理 |
| Postman / Apifox | 云函数本地调试、AI 接口联调 |
| ESLint + Prettier | 代码规范 |

## 四、技术要点

### 4.1 鉴权与数据隔离
- 所有请求依托 CloudBase 自动鉴权，前端调用 `wx.cloud.callFunction()` 时自动携带 openid
- 云函数内通过 `cloud.getWXContext().OPENID` 获取用户标识，无需通过参数传递
- 云数据库安全规则基于 `auth.openid` 做行级权限控制
- **禁止**前端直接传递 openid 作为查询条件（存在伪造风险）

### 4.2 图片上传流程
```
用户拍照/选图 → 前端压缩(wx.compressImage) 
→ 上传云存储(wx.cloud.uploadFile) → 获取 cloudID 
→ 写入 food_records 的 images 字段 → 渲染展示
```
- 单张图片压缩至 300KB 以内，控制流量
- 上传时携带 `cloudPath: food-images/{openid}/{timestamp}.jpg`
- 使用 `wx.cloud.getTempFileURL()` 获取临时 HTTPS 访问链接

### 4.3 AI 识别调用链路
```
前端上传图片 → 获取 cloudID → 调用云函数 aiRecognition
→ 云函数通过 cloud.downloadFile 下载图片 
→ 携带百度 API Key 请求菜品识别接口 
→ 解析返回的菜品名称、置信度 
→ 匹配营养数据库(内置FoodData)获取营养成分 
→ 返回结果给前端预览 → 用户确认/修正 → 写入云数据库
```

**安全要点**：百度 API 的 Secret Key **仅存储在云函数环境变量**中，不暴露到前端代码。

### 4.4 群组权限控制
- 群组采用「私密模式」：不对外公开，仅通过邀请卡片加入
- 邀请卡片携带群组 ID + 加密签名，云函数校验签名防篡改
- 群组动态流仅拉取成员中 `visibility: group` 的记录
- 云数据库安全规则确保跨群组不可见

### 4.5 分享卡片机制
```
分享方 → wx.shareAppMessage({
  path: '/pages/record/detail?id={recordId}',
  imageUrl: 自动生成分享卡片图
})
→ 接收方点击卡片 → 小程序 onLoad 解析 id 参数 
→ 校验权限 → 展示对应内容
```

### 4.6 性能优化要点
| 优化项 | 手段 |
|--------|------|
| 图片懒加载 | 列表页使用 `lazy-load` 属性 |
| 分页查询 | 云数据库 `.skip().limit(20)` 滚动分页 |
| 本地缓存 | 用户信息、当日记录缓存至 `wx.setStorage`，减少云调用 |
| 图片压缩 | 上传前 `wx.compressImage({ quality: 70 })` |
| 云函数冷启动 | 优化依赖包体积，避免引入大型 npm 包 |

### 4.7 数据结构关键设计
```
food_records 核心字段：
  - _openid: 自动维护的用户标识
  - date: 记录日期 (YYYY-MM-DD)
  - mealType: 餐次 (breakfast/lunch/dinner/snack)
  - content: 文字备注
  - images: [cloudID, ...]
  - aiResult: { foodName, confidence, ... }  // AI 识别结果
  - nutrition: { calories, carbs, protein, fat, fiber, sodium }
  - visibility: 'private' | 'group_xxx'  // 可见权限
```

## 五、技术选型决策

### 5.1 云开发 vs 自建后端

| 维度 | CloudBase 全托管 | 自建后端 (Node/Java) |
|------|-----------------|---------------------|
| 服务器 | 零服务器 | 需购买 + 运维 |
| 域名备案 | 不需要 | 需要 |
| 鉴权 | 自动对接微信登录 | 需自研 OAuth 流程 |
| 存储 | 自带 CDN 云存储 | 需对接 OSS/COS |
| 数据库 | NoSQL 免建表 | 需建表维护 |
| 成本 | 免费额度起步 | 服务器月费 50+ |
| 适配度 | 与微信原生能力无缝对接 | 需额外适配 |

**结论**：CloudBase 是唯一推荐方案，无替代方案更适配本项目。

### 5.2 AI 识别服务选型

| 维度 | 百度智能云 | 阿里云 | 火山引擎 |
|------|----------|--------|---------|
| 菜品识别准确率 | 高（中餐优化好） | 中高 | 中 |
| 基础免费额度 | 1000 次/月 | 1000 次/月 | 500 次/月 |
| 按量单价 | 0.004 元/次 | 0.0035 元/次 | 0.003 元/次 |
| SDK 成熟度 | 成熟 | 成熟 | 一般 |
| 文档质量 | 好 | 好 | 一般 |

**结论**：百度智能云为首选（中餐识别优化最好），阿里云为备选。

### 5.3 前端框架选型

| 维度 | 原生小程序 | Uni-app | Taro |
|------|----------|---------|------|
| 学习成本 | 低 | 中 | 高 |
| 包体积 | 最小 | 中等 | 较大 |
| 微信能力调用 | 原生无缝 | 需适配层 | 需适配层 |
| 跨端能力 | 仅微信 | H5 + 各平台 | H5 + 各平台 |
| 性能 | 最优 | 良好 | 良好 |

**结论**：原生小程序为首选。若后续需同步发布 H5 版本，可迁移至 Uni-app。

## 七、风险与注意事项

| 风险 | 应对策略 |
|------|----------|
| 微信审核不通过 | 遵循社交类目规范，不涉及敏感内容，首次提交尽量精简功能 |
| AI 识别率不足 | 保留手动修正入口，后续可建立自定义菜品营养库 |
| 云函数冷启动慢 | 使用云函数常驻实例（付费）或优化依赖体积 |
| 存储成本上涨 | 前端压缩图片、云存储设置生命周期自动清理过期文件 |
| 并发瓶颈 | CloudBase 自动扩缩容，前期无需担心 |
| openid 泄露 | 云数据库安全规则禁止前端按 openid 查询，所有查询走云函数 |

## 八、开发规范速查

1. **命名规范**：文件/目录用小写下划线（`food_records`），云函数用小写驼峰（`aiRecognition`）
2. **Git 分支**：`main` 主线 / `dev` 开发分支 / `feature/xxx` 功能分支
3. **提交规范**：`feat:` 新功能 / `fix:` 修复 / `refactor:` 重构 / `docs:` 文档
4. **云函数部署**：每次修改后在微信开发者工具中右键上传，确保环境变量同步
5. **数据库索引**：`food_records` 的 `_openid + date` 建联合索引，提升查询性能
